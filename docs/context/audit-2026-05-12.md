## 1. The Safari blank-page bug (highest user-facing severity)

### Symptom
Some `/holidays/[slug]` pages render only the navbar and footer in Safari (and reportedly some other non-Chromium browsers). Reloads sometimes fix it. Chrome is unaffected.

### Why this happens

The app is configured for **streaming SSR with strict caching**:

- `next.config.js:48` — `cacheComponents: true`
- `next.config.js:62` — `experimental.dynamicIO: true`
- `next.config.js:49` — `reactCompiler: true`
- `app/layout.tsx:197-206` — Header / `{children}` / Footer all wrapped in their own `<Suspense>` boundaries

Streaming relies on the proxy flushing HTTP chunks the moment they're emitted. Your `infra/nginx/snippets/proxy-common.conf` is missing the streaming-safe directives, and `nginx.conf:28-51` aggressively gzips HTML.

Three specific nginx problems combine:

1. **`proxy_buffering` is implicitly ON (the nginx default).** Next.js writes a Suspense chunk; nginx buffers it; if the buffer never fills and the connection idles, nginx flushes nothing. Safari's strict streaming HTML parser receives a `<head>`+navbar+footer shell and then a long pause, and the page-load lifecycle declares the body "complete" before the suspended content arrives.
2. **`gzip on` + `gzip_proxied any` (`nginx.conf:28-30`)** applied to streamed `text/html`. gzip needs to flush its block boundary — with buffering on and small chunks, the gzip flush rarely lines up with Suspense boundaries, so Safari sees an entirely truncated body.
3. **`proxy_set_header Connection "upgrade"` unconditionally** (`proxy-common.conf:9`). The `$connection_upgrade` map defined at `nginx.conf:57-60` is *not* being used. Setting `Connection: upgrade` on every regular HTTP request confuses some HTTP/2 clients (Safari especially) and can cause them to close the connection mid-stream.

Chrome's HTML parser is more forgiving about delayed Suspense chunks and recovers; Safari/WebKit drops the un-streamed content.

A real secondary path: `formatWeekdayAr()` in `lib/holidays/page-data.js:53` uses `Intl.DateTimeFormat('ar-SA-u-nu-latn', ...)` — older Safari/iOS versions do not understand the `nu-latn` extension. It is wrapped in try/catch so it does not throw, but if it ever did, you would see exactly this symptom because the throw happens *after* the shell has already streamed.

### Fix

**`infra/nginx/snippets/proxy-common.conf`** — replace with:

```nginx
proxy_http_version 1.1;

proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;

# Use the map defined in nginx.conf — only set "upgrade" for actual WebSocket requests
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection $connection_upgrade;

# Streaming-safe proxy settings (Next.js 16 RSC + Suspense)
proxy_buffering off;
proxy_request_buffering off;
proxy_read_timeout 75s;
proxy_send_timeout 75s;
```

**`infra/nginx/nginx.conf`** — turn gzip OFF for streamed HTML or move it behind `gzip_types text/css application/javascript application/json image/svg+xml` only. The simplest safe change:

```nginx
gzip on;
gzip_vary on;
gzip_proxied any;
gzip_comp_level 5;
gzip_min_length 1024;
gzip_types
  text/css
  text/xml
  text/javascript
  application/json
  application/javascript
  application/xml
  application/xml+rss
  application/atom+xml
  application/ld+json
  image/svg+xml
  font/woff
  font/woff2;
# NOTE: text/html removed. Let Next.js / the upstream decide HTML compression
# (it sets Content-Encoding when appropriate). gzip + streaming HTML breaks Safari.
```

Add an HTTPS server block to `templates/site.conf.template` — right now only `listen 80;` exists. If TLS is terminated upstream (Cloudflare in front), document it. Otherwise users hitting `https://` cannot reach the app at all.

After deploying, validate from a Mac:

```bash
curl -v --http1.1 -H "Accept-Encoding: identity" https://miqatona.com/holidays/ramadan 2>&1 | head -40
curl -v --http2 -H "Accept-Encoding: gzip" https://miqatona.com/holidays/ramadan 2>&1 | head -40
```

Look for `Transfer-Encoding: chunked`, no early `Content-Length`, and progressive output.

### Defensive code change

`src/lib/holidays/page-data.js:53` — replace the locale string with one that works everywhere:

```js
function formatWeekdayAr(date) {
  try {
    return new Intl.DateTimeFormat('ar', { weekday: 'long' }).format(date);
  } catch {
    return '';
  }
}
```

`src/components/clocks/CountdownTicker` — verify it has a server-rendered fallback (the static remaining-days number) so even if the client island fails to hydrate, the SSR HTML still shows the counter. The page already passes `initialRemaining`; just make sure no `useLayoutEffect`/`window` access happens before mount.

## 2. Server errors on `*/opengraph-image` (17 URLs in GSC)

### Why it happens

`src/lib/db/queries/countries.ts:100-112`:

```ts
export async function getCountryBySlug(slug: string): Promise<Country | null> {
  'use cache'
  cacheTag('countries', `country-${slug}`)
  cacheLife('days')

  const fromFallback = findFallbackCountryBySlug(slug)
  if (fromFallback) return fromFallback

  try {
    return await fetchCountryBySlugFromDb(slug)
  } catch {
    throw new Error(`Country not found: ${slug}`)   // ← THIS LINE
  }
}
```

The return type promises `Country | null`, but the catch turns DB errors *and* "not found" into a thrown exception. The sibling `getCountryByCode()` at line 115 correctly returns `null` on the same failure. Inconsistent contract.

`src/app/time-now/[country]/opengraph-image.jsx:59` defends with `.catch(() => null)`, so this particular route returns the fallback image — but Google has indexed paths like `/awqaf-al-salat/...` (typo from a previous version's redirect chain) and `/time-now/eqypt/opengraph-image` where the OG handler is hit before the parent page returns 404. The `'use cache'` boundary can also rethrow inside cache evaluation in some Next 16 paths, so `.catch` does not always catch in production.

The bigger issue is policy: **OG image routes should never 5xx — they should always render a fallback.**

### Fix

1. **`src/lib/db/queries/countries.ts:111`** — change `throw new Error(...)` to `return null` and let callers decide. This single line resolves the largest share of the 17 5xx reports.

2. **`src/app/holidays/[slug]/opengraph-image.jsx:52`** — `searchParams` is now a Promise in Next 15+/16 server-side:

   ```jsx
   export default async function Image({ params, searchParams }) {
     const { slug } = await params;
     const sp = (await searchParams) || {};
     const isSquare = sp.sq === '1';
     // ...
   }
   ```

   Right now line 56 does `searchParams?.sq === '1'` directly on a Promise → always `undefined` → never produces the square variant, *and* in dev this logs an async-API warning that becomes a hard error if anything else awaits incorrectly.

3. **Wrap every `opengraph-image.*` `export default` in a top-level try/catch** that always falls back to a branded image. Pattern:

   ```jsx
   export default async function Image(args) {
     try {
       return await renderRealImage(args);
     } catch (err) {
       logEvent('og-image-fallback', { err: String(err) });
       return renderFallbackImage();
     }
   }
   ```

   Files to update:
   - `src/app/holidays/[slug]/opengraph-image.jsx`
   - `src/app/time-now/[country]/opengraph-image.jsx`
   - `src/app/time-now/[country]/[city]/opengraph-image.jsx` (if present)
   - `src/app/mwaqit-al-salat/[country]/opengraph-image.jsx` (if present)
   - `src/app/mwaqit-al-salat/[country]/[city]/opengraph-image.jsx` (if present)

4. **Decide policy for unknown OG slugs**: today `next/og` will happily render a fallback for `/holidays/madrid-in-usa/opengraph-image` even though the parent page is a 404. Either:
   - Make the OG route call `notFound()` when the slug is invalid (so Google records a 404 and de-indexes it), or
   - Always return a 200 generic fallback PNG and let `/holidays/[slug]` itself 404 the parent URL. Pick one and apply across the four routes above. The 200-fallback approach matches Vercel's default behavior and is the safer choice for Search Console.

5. **Arabic glyphs**: `next/og` does not have Arabic fonts in its system fallback. Every event title in `holidays/[slug]/opengraph-image.jsx` is Arabic. The current implementation relies on `fontFamily: 'system-ui,-apple-system,sans-serif'` which often produces tofu boxes inside the rasterizer. Load an Arabic font and pass it through `ImageResponse.options.fonts`:

   ```jsx
   const arabicFont = await fetch(new URL('../../assets/NotoSansArabic-Bold.ttf', import.meta.url))
     .then(r => r.arrayBuffer());
   return new ImageResponse(<JSX/>, {
     width: W, height: H,
     fonts: [{ name: 'NotoArabic', data: arabicFont, style: 'normal', weight: 800 }],
   });
   ```

   Without this, share-card previews on WhatsApp/Twitter look broken even when the route does not 5xx.

## 3. Missing `performer` field on Event structured data

### Why it happens

`src/lib/holidays-engine.js:529-583` — `buildEventSchema()` emits `name`, `description`, `startDate`, `endDate`, `eventStatus`, `eventAttendanceMode`, `location`, `organizer`, `audience`, `image`. **`performer` is missing.** Schema.org / Google's `Event` rich-result requirements list `performer` as a "recommended" field; Google flags its absence as a warning.

The four flagged events are `school-start-kuwait`, `liberation-day-kuwait`, `green-march-morocco`, `independence-day-tunisia`. These are **public observances, not bookable events** — they should not be emitting `Event` structured data at all. The guard at `lib/holidays/page-data.js:173` (`shouldEmitEventSchema = pageSchemaEvent?.schemaData?.googleEventRichResult === true`) is correct in spirit, but if any of those four events' `package.json` has `schemaData.googleEventRichResult: true`, the gate lets them through. Verify:

```bash
for f in school-start-kuwait liberation-day-kuwait green-march-morocco independence-day-tunisia; do
  echo "=== $f ==="
  grep -A2 googleEventRichResult src/data/holidays/events/$f/package.json
done
```

### Fix

Two-part:

1. **Add `performer` to `buildEventSchema`** so that any event that *does* legitimately emit `Event` schema passes Google validation. Insert after the `organizer` block (line 577):

   ```js
   performer: {
     '@type': 'Organization',
     name: schemaData.eventName || ev.name,
   },
   ```

   For Islamic events you can use the event name as performer; for national days, the country name. The cleanest pattern is "the observance is performed by the people of the country", so use the audience.area or country name.

2. **Remove the Event schema entirely from the 4 flagged events** by setting `schemaData.googleEventRichResult: false` (or leaving it undefined) in their package.json files. These pages are countdowns, not bookable events. A `WebPage` + `FAQPage` + `BreadcrumbList` graph is sufficient and is what Google actually rewards for "how long until..." queries.

3. **Tighten the gate**: in `lib/holidays/page-data.js:173`, default to *false* and require an explicit opt-in. Already implemented correctly with `=== true`, but document it in `docs/events-authoring-guide.md` so new content authors don't accidentally turn it on.

## 4. Duplicate `FAQPage` schema (and other JSON-LD hygiene)

### Why GSC reports "Duplicate field FAQPage"

GSC's wording is misleading: the URL `https://miqatona.com/holidays` appears twice in the affected-items table because **two separate FAQPage payloads are emitted on that URL.** Today the `/holidays` page itself only emits *one* FAQPage (via `components/holidays/GlobalSchemas.jsx:93-98`). Two possibilities:

a) **A previous deploy had two.** The comment at `src/app/holidays/page.jsx:135-137` is a guard against re-introducing it. Verify Google's stored snapshot via *URL Inspection → View crawled page* — if the rendered HTML there shows only one `<script type="application/ld+json">` containing `"@type":"FAQPage"`, hit *Validate Fix* in GSC. Google will re-crawl and the issue will clear within ~7 days.

b) **Server is returning a cached HTML that still has two.** `next.config.js:96-107` caches `/holidays/:path*` with `s-maxage=86400, stale-while-revalidate=3600`. After fixing the underlying duplicate you must purge the CDN/nginx cache (or wait 24h) before requesting validation.

### Other JSON-LD hygiene problems found

These are not strictly "duplicate FAQPage" but they will eventually trip the same warnings:

- **`src/app/date/calendar/[year]/page.tsx:134-152`** — emits a `WebPage` whose `mainEntity` is a `FAQPage`. Schema.org allows nesting, but Google's parser prefers FAQPage as a *top-level* `<script type="application/ld+json">` payload. Split them:

  ```jsx
  <JsonLd data={webPageSchema} />
  <JsonLd data={faqSchema} />
  ```

- **`src/components/holidays/GlobalSchemas.jsx`** emits 4 schemas (WebPage, BreadcrumbList, FAQPage, ItemList) and **`src/app/holidays/page.jsx:104-147`** *also* emits a `CollectionPage` whose `mainEntity` is an `ItemList`, plus a standalone `ItemList`, plus a `WebSite`, plus an `Organization`, plus a `BreadcrumbList`. That is **6 schemas on `/holidays`, two of them ItemList, both BreadcrumbList competing**, and `WebSite` is *already* emitted globally by `components/seo/SiteWideSchemas.jsx`. Consolidate:

  Files to deduplicate on `/holidays`:
  1. Drop the WebSite + Organization graphs from `app/holidays/page.jsx:53-71` — they are already in `SiteWideSchemas` rendered by the root layout.
  2. Keep only one BreadcrumbList — prefer the one in `GlobalSchemas`, drop the one inline in `page.jsx:49-52`.
  3. Keep only one ItemList — prefer `holidaysCollectionSchema` (it's wrapped in `CollectionPage` which is semantically correct), drop the standalone `upcomingEventsSchema` at `page.jsx:123-133`.
  4. Result: `/holidays` has *one* WebPage, *one* BreadcrumbList, *one* FAQPage, *one* CollectionPage. Clean.

- **Per-slug `/holidays/[slug]/page.jsx`** at lines 107-112 emits up to 6 schemas. Verify they all carry unique `@id` or are clearly distinguishable. `buildEventSchema` and `buildEventSeriesSchema` both describe the same entity and can confuse Google's entity extraction. Decide: pick `Event` OR `EventSeries` per page, not both.

## 5. 431 noindex'd pages + 548 "discovered, not indexed"

### Root cause #1 — sitemaps contradict the page-level `noindex`

`/date/calendar/[year]/page.tsx:48` decides indexability with `isSeoIndexableGregorianCalendarYear(numericYear, currentYear)` — typically a window like `currentYear ± 2`. But `generateStaticParams` (line 22) emits 11 years (`-5` to `+5`). The 6+ years outside the indexable window get `index: false` *and* are included in the sitemap → GSC fills the "Excluded by noindex" bucket.

Same shape for `/date/hijri/[year]/[month]/[day]/page.tsx` and `/date/[year]/[month]/[day]/page.tsx` — per `docs/platform-stabilization-2026-q2.md` these should be `noindex, follow` for the bridge, *and removed from sitemaps*.

### Root cause #2 — sitemap fan-out without internal-link support

The 548 "Discovered – currently not indexed" includes `/calculators/end-of-service-benefits`, `/calculators/vat`, daily calendar pages, etc. These are *announced* (sitemap) but Google doesn't see enough internal links, content depth, or quality signals to budget a crawl.

### Fix

1. **Filter sitemaps by the page's own `robots.index`.** Each `sitemap.js` / `sitemap.xml/route.ts` should call the same `isSeoIndexable*()` helpers the page uses, and only emit URLs where the helper returns `true`. Concretely:
   - `src/app/date/sitemaps/calendars/route.ts` — gate every year through `isSeoIndexableGregorianCalendarYear` / `isSeoIndexableHijriCalendarYear`.
   - `src/app/date/sitemap.xml/route.ts` and `src/app/date/sitemaps/*` — remove all daily date URLs entirely (they are `noindex` per the platform plan).
   - `src/app/holidays/sitemap.js` — only include events where `getEventMeta(slug)?.publishStatus === 'live'`.

2. **Drop dead sitemap routes.** `/date/gregorian/sitemap.xml` and `/date/hijri/sitemap.xml` are intentionally empty per the docs — empty sitemaps confuse GSC. Either delete the route entirely or have them return `404`.

3. **Add internal links to thin-content calculator pages.** Each `/calculators/<tool>/page.jsx` should link to 5-8 sibling tools and to a parent "/calculators" hub. Use the existing `GeoInternalLinks` / `appendToolDiscoveryLinks` pattern from `/holidays/page.jsx:81-97`. Without backlinks, Google will keep saying "Discovered, not indexed."

4. **Reconcile `mwaqit-al-salat` typo URLs.** GSC examples include `/awqaf-al-salat/saudi/jeddah/opengraph-image` — that path does not exist in the codebase. These are Google revisiting URLs from before the rename. Add a `next.config.js` redirect:

   ```js
   { source: '/awqaf-al-salat/:path*', destination: '/mwaqit-al-salat/:path*', permanent: true },
   ```

   Place it near the existing `/prayer-times/:country/:city` redirect at line 197.

## 6. The `/&` redirect error

`next.config.js:182-185` already redirects `/&` → `/`. GSC still flags it because the bad URL is `https://www.miqatona.com/&` (note `www`). The first redirect on line 176-180 rewrites `www.miqatona.com/:path*` → `https://miqatona.com/:path*`, so `/&` is preserved on the apex. Two issues:

1. Some user agents URL-encode `&` to `%26` before requesting; the source matcher needs both forms.
2. The redirect lands at `/` with no further redirect — fine — but if a query string `?utm_source=foo&` slipped into the path during a previous bad share, the path may be `/&utm_source=foo` and not match the static `/&`.

### Fix

Replace the single `/&` rule with:

```js
{
  source: '/:path*',
  has: [{ type: 'query', key: 'utm_source' }],
  missing: [{ type: 'header', key: 'x-skip' }],   // tag to avoid loop
  destination: '/:path*',
  permanent: false,
},
{ source: '/&', destination: '/', permanent: true },
{ source: '/%26', destination: '/', permanent: true },
{ source: '/&:rest*', destination: '/', permanent: true },
```

Or — simpler and safer — add a top-level `middleware.ts` that normalizes weird paths:

```ts
import { NextResponse } from 'next/server';
export function middleware(req) {
  const url = new URL(req.url);
  if (url.pathname === '/&' || url.pathname.startsWith('/&')) {
    return NextResponse.redirect(new URL('/', url), 308);
  }
}
```

After deploying, click "Validate Fix" in GSC for the redirect error.

## 8. Caching architecture — `cacheComponents` + `dynamicIO` ⚠

This is the single biggest *architectural* risk in the migration. Both flags are experimental in Next 16.1.6 and they change the semantics of every server-component data read.

### The contract

- Any server component / function that reads "dynamic" data (DB, current time, headers, cookies, searchParams) must either:
  - be inside a `'use cache'` boundary with explicit `cacheTag` + `cacheLife` / `cacheLife({ revalidate, expire })`, **or**
  - be inside a `<Suspense>` boundary so PPR can stream the dynamic part separately.

### Where the contract is violated

- **`src/lib/holidays/page-data.js:99-264`** is one big `'use cache'` block that fetches *everything* — event data, current time-remaining, schemas, etc. It calls `getTimeRemaining(targetDate, nowMs)` and similar. The cache key is just `slug`, but the *result* depends on `Date.now()` (via `nowMs` inside `resolveHolidayRuntimeData`). This means: the first crawler to hit the page caches a value of `remaining.days` that is then served stale for the `cacheLife('hours')` window. SSR and the client tick will disagree by hours → hydration mismatch → blank content (root cause candidate #2 for the Safari bug).

  **Fix:** split the function. Cache the *static* parts (event content, FAQ, schemas with placeholder dates) with `'use cache'`. Resolve the *time-dependent* parts (`targetDate`, `remaining`, `eventState`, `gregStr`, `hijriStr`) outside the cache, freshly per request, and pass them via props to the cached children. Pattern:

  ```js
  // page.jsx
  const staticData = await getHolidayStaticData(slug);      // 'use cache'
  const liveData = await getHolidayLiveData(slug);          // dynamic, inside <Suspense>
  ```

- **`src/lib/date-utils.ts` `getCachedNowIso()`** — 44 callers per the agent audit. If this is `'use cache'`, every caller gets a stale "now". Either rename to make the staleness explicit (`getHourBucketIso()`) or stop caching it. Most "remaining days" math needs *real* current time, not a cached one.

- **`/holidays` page** mixes `getCachedNowIso()` (cached) with `new Date().getFullYear()` in the same metadata function. The year inside the title can flip on Jan 1 before the cache flushes.

### Audit checklist

Search the codebase for these violations:

```bash
grep -rn "new Date()" src/app src/lib | grep -v "/scripts/" | grep -v ".test."
grep -rn "Date.now()" src/app src/lib | grep -v "/scripts/" | grep -v ".test."
grep -rn "'use cache'" src
```

Anywhere `new Date()` or `Date.now()` appears inside a `'use cache'` function, the result is non-deterministic and breaks PPR. Either pull the time read out, or accept the cached freeze and document it.

## 9. Architecture & clean-code issues

Ranked by impact. Not all are bugs — several are sources of future bugs.

### 9.1 Dual data layer for holidays (high)

- `src/lib/holidays-engine.js` (727 LOC) — legacy god-file. Exports `ALL_EVENTS`, `ALL_EVENT_SLUGS`, `buildEventSchema`, `buildFAQSchema`, helpers.
- `src/lib/holidays/repository.js` + `runtime-data.js` + `page-data.js` + `metadata.js` + `og-data.js` + `display.js` + `taxonomy.js` + `country-dates.js` — the newer modular boundary.

Both are imported live (`/holidays/[slug]/page.jsx:15-26`). The engine has not been deprecated. New contributors will not know which to use.

**Fix path:** make `holidays-engine.js` a *re-export shim* — every named export delegates to the new modules. Add a JSDoc deprecation banner. Schedule the engine for deletion once nothing imports it directly.

### 9.2 Mixed JS / TS (medium)

`src/lib` has `.js`, `.jsx`, `.ts`, `.tsx` files side by side. `src/lib/holidays/page-data.js` is `.js`; `src/lib/holidays/types.ts` is `.ts`. This causes:

- No type-checking on the `.js` files unless `checkJs` is on
- Discovery of the right file to edit takes longer
- IDE auto-import behaves unpredictably

**Fix path:** make `tsconfig.json` enforce `allowJs: true, checkJs: false` (current state) but add a project-level rule: all new files go `.ts/.tsx`. Migrate `holidays/page-data.js` → `.ts` first (it's central) and let the type errors guide cleanup.

### 9.3 Inconsistent route conventions (medium)

`src/app/holidays/` has `page.jsx`, `loading.jsx`, `not-found.jsx` (all `.jsx`).
`src/app/date/` has `page.tsx`, `loading.tsx`, `not-found.tsx` (all `.tsx`).
`src/app/calculators/` mixed.

Inside the holidays `[slug]/` folder, the file naming is fine. But site-wide it's chaotic.

### 9.4 `mwaqit-al-salat` route naming (low — but SEO-relevant)

The chosen URL `mwaqit-al-salat` is a transliteration of "مواقيت الصلاة". Google indexes it. But the GSC error logs show `awqaf-al-salat` (typo) — that means *somewhere*, possibly in an old version or in a 3rd-party share, the typo URL escaped. Add the redirect (section 5 fix #4) and accept that the URL is what it is. Renaming now would erase all current rankings — *do not rename*.

### 9.5 Sitemap files do not share a base helper (medium)

8 separate `sitemap.js` / `sitemap.xml/route.ts` files, each duplicating header logic, lastmod stamping, filtering. Extract:

```ts
// src/lib/seo/sitemap-builder.ts
export function buildSitemap(entries: SitemapEntry[]): MetadataRoute.Sitemap { ... }
export function withCacheHeaders(res: Response): Response { ... }
```

…and have every sitemap route call it. Right now drift between routes (different `changefreq`, different `priority`) is a real signal-noise problem for Google.

### 9.6 Inline styles everywhere (low)

`src/app/holidays/[slug]/page.jsx` has ~30 inline `style={...}` objects. They reference design tokens (`var(--space-2)`, etc.), so they're not arbitrary, but they bloat the SSR HTML and make the page hard to read. Move to CSS classes (you already have a design-system pattern with `.card`, `.btn`, etc.).

### 9.7 `dynamic(() => import(...))` for SSR-needed sections (medium)

`src/components/holidays/index.jsx:22-27` uses `dynamic(() => import('./SectionFAQ'), { ssr: true })` for sections that should be in the initial HTML for SEO. With `ssr: true` the SSR cost is unchanged; this purely adds a client chunk split which is fine. But:

- The `Suspense` wrapper around each section streams empty `<div>` fallbacks (`SectionSkeleton`). For Safari (see section 1), these empty placeholders can replace real content if the stream is interrupted.
- **For SEO-critical content like FAQ, prefer a *static* import + plain `<Suspense>`, not `dynamic()`.** Reserve `dynamic()` for genuinely below-the-fold interactive content.

### 9.8 `useClient` boundary leaks (low — verify)

`src/components/seo/SiteWideSchemas.jsx` is a *server* component that emits JSON-LD. Good. But `src/components/ErrorBoundary.client.jsx` wraps `{children}` in `app/layout.tsx:200` — this means every render goes through a client boundary. If any child throws during SSR (e.g. `Intl.DateTimeFormat` issue from section 1), the boundary catches it client-side but the SSR HTML already streamed up to the throw point. That's exactly the "navbar + footer only" symptom.

**Fix:** add a *server-side* error boundary using Next.js's built-in `error.tsx` per route segment. `src/app/error.tsx` exists — verify it actually renders something useful, not just `null`.

### 9.9 Prisma config (low)

- `prisma.config.ts` exists alongside `prisma/schema.prisma` — Next.js + Prisma in standalone mode can lose the schema unless `output = "../src/generated"` is set and the generated client is bundled. Verify your Dockerfile copies `src/generated/` into the runtime image.
- Add indices on lookup columns: `City.citySlug`, `Country.countrySlug`, `City.country_code`. Without them, country dropdowns on every page hit a sequential scan.

### 9.10 Tests do not cover the migration (medium)

`tests/` has unit tests for SEO helpers and holidays formatters. **None** test:

- `getHolidayPageData` end-to-end with `'use cache'` semantics
- Sitemap output (URL list, lastmod, robots compatibility)
- OG image fallback behavior on unknown slugs
- The Prisma adapter actually connects with the configured DATABASE_URL

Add at minimum:

```ts
// tests/sitemap.test.ts — assert no noindex'd URL appears in any sitemap
// tests/og-fallback.test.ts — assert OG routes never throw on unknown slugs
// tests/event-schema.test.ts — assert performer is always present when Event is emitted
```

### 9.11 Header / Footer in `<Suspense>` with fixed-height fallback (low)

`layout.tsx:197` wraps `<Header />` in `<Suspense fallback={<div className="h-16" />}>`. This is a layout-shift fix, but `<Header />` is a *static* component (no async data). Wrapping it in Suspense for nothing forces the React reconciler to manage an extra boundary and adds noise to the SSR stream. Drop the Suspense around Header and Footer unless they really fetch data.

### 9.12 React Compiler enabled with experimental flags (low)

`next.config.js:49` `reactCompiler: true` combined with `cacheComponents: true` and `dynamicIO: true` is three experimental features stacked. Any one of them changing semantics across a Next.js minor bump can cause regressions. Pin the Next.js version exactly (`16.1.6`, not `^16.1.6`) and add a comment in `package.json` warning about the experimental dependency.
